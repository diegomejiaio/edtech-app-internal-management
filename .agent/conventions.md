## 2. Cosmos DB

- Account: shared `shared-cosmos-nosql` in `rg-shared-services`.
- Logical DBs: `espaciopro` (prod), `espaciopro-dev` (local dev).
- Containers: `master` (Catalogs, Students, Teachers, Schedules), `operations` (Enrollments, StudentPayments, TeacherPayments, Expenses).
- **Partition key**: `/type` for both containers.
- **ID**: `Guid.NewGuid().ToString()`. No surrogate keys.
- **Document shape** (every doc):
  ```json
  {
    "id": "<guid>",
    "type": "<EntityName>",
    "createdAt": "<iso>",
    "createdBy": { "clerkUserId": "...", "email": "...", "displayName": "..." },
    "updatedAt": "<iso>",
    "updatedBy": { ... },
    "deletedAt": null,
    "deletedBy": null,
    ...entityFields
  }
  ```
- **Soft delete**: set `deletedAt` + `deletedBy`. Never `DeleteItemAsync`.
- **Queries** must filter `WHERE c.type = @type AND (c.deletedAt = null OR NOT IS_DEFINED(c.deletedAt))` unless explicitly listing deleted.
- **Unique keys**: defined at container creation, **immutable**. Strategy for `(docType, docNumber)` to be decided in M0 (paths A/B/C documented in `docs/02-architecture.md`).

## 3. Audit

- Every entity inherits `BaseEntity` (audit fields).
- `AuditUser` is a value object snapshot: `{ clerkUserId, email, displayName }` taken at write time. Never resolved by lookup.
- Repositories receive `ICurrentUser` (current Clerk user) injected per-request. They auto-populate `createdBy/updatedBy/deletedBy`.
- **Do not** mutate audit fields manually in business code.

---

## 4. Auth

- Clerk JWT (RS256) validated against Clerk's public JWKS endpoint. No Secret Key in v1.
- JWKS cached in-memory (helper `ClerkJwtValidator`).
- Required claims in JWT template (configured in Clerk Dashboard): `sub`, `email`, `name`, `role` (`{{user.public_metadata.role}}`).
- **v1 role**: only `admin` is enforced. Reject all other roles with 403.

---

## 5. API

- HTTP triggers, REST style, JSON only.
- Routes: `/api/v1/{resource}`.
- Auth: `Authorization: Bearer <jwt>` header. Validation in middleware.
- DTOs: separate from domain entities. Map at controller boundary.
- Errors: RFC 7807 Problem Details JSON.

---

## 6. Frontend

- Next.js 16 with `output: 'export'`. **No SSR, no API routes**.
- All data fetching client-side via `fetch` to backend Functions.
- Auth: Clerk's React SDK. JWT extracted client-side and attached as `Bearer`.
- UI strings: Spanish. Hardcoded in components is OK for v1 (no i18n framework).
- Components: container/presentational pattern. Atomic design when complexity grows.

## 7. Workflow

- **Specifications**: use Speckit (`.specify/` directory). Spec-first for any non-trivial feature.
- **Maintenance stories**: generated by HVE upstream (`lib/hve-core` submodule). This repo is a consumer.
- **OpenSpec**: NOT used. Do not propose it.
- **SDD orchestrator** (`/sdd-*`): NOT initialized in this repo. Decision postponed.
- **Skills** (`go-testing`, `skill-creator`, etc.): load BEFORE writing code when context matches.

## 8. Git & Commits

- Conventional Commits format: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`.
- **NEVER** include `Co-Authored-By` or AI attribution.
- One logical change per commit.
- Branch off `main`. PRs preferred; direct push allowed for solo work.
